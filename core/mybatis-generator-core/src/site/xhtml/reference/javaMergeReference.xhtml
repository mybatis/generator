Java Merge Reference

Java code merging is a complex operation and there are tradeoffs to consider. The Java merger is highly configurable. This page will provide some details to help you decide which configuration is best for your needs.

The Java merger is built using the parser from https://javaparser.org/. It is possible that future changes in that product may change some of what's written here. We will endeavor to keep this page up to date as things change.

Any configuration of the Java merger will preserve the function of the merged Java code. Merge configuration is primarily concerned with formatting of the code after the merge operation

Configuration options for the parser include the following broad topics:

  • Lexical Preservation
  • Merge Strategy
  • Import Sorting
  • Indentation Options

We will cover these in order.

Recommendations

The default merge configuration is:

  1. Lexical preservation: disabled
  2. Merge strategy: MERGE_INTO_EXISTING
  3. Import sorting: ECLIPSE
  4. Indentation: 4 spaces

In most cases, the default configurtion will work. The limitations of the default configuration are:

  • Because lexical preservation is disabled, existing code may be reformatted.
  • Class/Interface to record merges are not supported

All strategies have the most trouble with comments. Non-JavaDoc comments will be completely lost if you use the "merge into new" strategy. If you wish to have the code formatting benefits of the lexical preserving printer, then you should be very cautious with commnets (both Javadoc and non-Javadoc comments).

Lexical Preservation

Lexical preservation is a function of JavaParser whereby the formatter (or "Printer") of the merged code attempts to preserve the lexical structure and format of merged code. This is a highly complex process, and it does not always work flawlessly. But it may be useful if you have added large amounts of custom code to a generated Java file, and you wish to preserve the format of the custom code. A good example is when you have a method that uses stream operations. Suppose you have a method like the following:

public int sumOfEvens(int n) {
    return IntStream.of(0, n).filter(i -> i % 2 == 0).map(i -> i * 2).sum();
}

The lexical preserving printer will keep the format of this code. If you don't use the lexical preserving printer, then the code may end up like this: 

public int sumOfEvens(int n) {
    return IntStream.of(0, n)
                    .filter(i -> i % 2 == 0)
                    .map(i -> i * 2)
                    .sum();
}

Limitations of the Lexical Preserving Printer

  • If you choose lexical preservation, then any indentation option you configure will be ignored. The lexical preserving printer attempts to match the indentation of the original code.
  • If you choose lexical preservation, then any import sorting option you configure will be ignored. The lexical preserving printer does not sort or change the order of imports.
  • The lexical preserving printer has challenges with comments. In some cases, JavaDoc comments are not successfully merged.

Merge Strategy

There are two possible merge strategies:

  • Merge newly generated code into existing code (MERGE_INTO_EXISTING)
  • Merge existing code into newly generated code (MERGE_INTO_NEW)

The basic goal of the merger - in either case - is to preserve custom methods, fields, inner classes, etc. from an existing Java file. The difference is the starting point.

The MERGE_INTO_EXISTING strategy works like this:

  1. Remove all generated items from an existing file
  2. Copy generated items from a new file into the existing file
  3. Return the (merged) existing file

The MERGE_INTO_NEW strategy works like this:

  1. Copy custom items from an existing file into the new file
  2. Return the (merged) new file
  3. This strategy will NOT copy non-Javadoc comments from existing files into a new file

In many cases the outcomes will be the same regardless of strategy. The differences are detailed below:

  • The MERGE_INTO_NEW strategy will not copy non-JavaDoc comments from existing code.
  • The MERGE_INTO_NEW strategy will perform a Class or Interface to Record Conversion. This is because the merger does not change the generated type (class, interface, record, etc.) to match the existing type.

Import Sorting

Import sorting options are ignored if you chose the lexical preserving option.

If you do not choose the lexical preserving option, then there are three options for soring imports. These options are built in to JavaParser - we do not create any custom sort options. The options are as follows:

ECLIPSE
Groups of static imports first, then java.*, then javax.*, then org.*, then com.*, others
INTELLIJ
Groups of other imports first, the java.* and javax.*, then static imports
DEFAULt
All imports in the same group

All groups are sorted alphabetically.

Indentation Options

The indentation options (tabs vs. spaces, and size) specified in the generator configuration are also used by the merger – but only if you are not using the lexical preserving printer.
