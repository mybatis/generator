29  /**
30   * This interface defines methods that will be called at different times during
31   * the code generation process. These methods can be used to extend or modify
32   * the generated code. Clients may implement this interface in its entirety, or
33   * extend the PluginAdapter (highly recommended).
34   * <p>
35   * Plugins have a lifecycle. In general, the lifecycle is this:
36   * 
37   * <ol>
38   * <li>The setXXX methods are called one time</li>
39   * <li>The validate method is called one time</li>
40   * <li>The initialized method is called for each introspected table</li>
41   * <li>The clientXXX methods are called for each introspected table</li>
42   * <li>The providerXXX methods are called for each introspected table</li>
43   * <li>The modelXXX methods are called for each introspected table</li>
44   * <li>The sqlMapXXX methods are called for each introspected table</li>
45   * <li>The contextGenerateAdditionalJavaFiles(IntrospectedTable) method is
46   * called for each introspected table</li>
47   * <li>The contextGenerateAdditionalXmlFiles(IntrospectedTable) method is called
48   * for each introspected table</li>
49   * <li>The contextGenerateAdditionalJavaFiles() method is called one time</li>
50   * <li>The contextGenerateAdditionalXmlFiles() method is called one time</li>
51   * </ol>
52   * 
53   * Plugins are related to contexts - so each context will have its own set of
54   * plugins. If the same plugin is specified in multiple contexts, then each
55   * context will hold a unique instance of the plugin.
56   * <p>
57   * Plugins are called, and initialized, in the same order they are specified in
58   * the configuration.
59   * <p>
60   * The clientXXX, modelXXX, and sqlMapXXX methods are called by the code
61   * generators. If you replace the default code generators with other
62   * implementations, these methods may not be called.
63   * 
64   * @author Jeff Butler
65   * @see PluginAdapter
66   * 
67   */

87      /**
88       * This method is called just before the getGeneratedXXXFiles methods are
89       * called on the introspected table. Plugins can implement this method to
90       * override any of the default attributes, or change the results of database
91       * introspection, before any code generation activities occur. Attributes
92       * are listed as static Strings with the prefix ATTR_ in IntrospectedTable.
93       * <p>
94       * A good example of overriding an attribute would be the case where a user
95       * wanted to change the name of one of the generated classes, change the
96       * target package, or change the name of the generated SQL map file.
97       * <p>
98       * <b>Warning:</b> Anything that is listed as an attribute should not be
99       * changed by one of the other plugin methods. For example, if you want to
100      * change the name of a generated example class, you should not simply
101      * change the Type in the <code>modelExampleClassGenerated()</code> method.
102      * If you do, the change will not be reflected in other generated artifacts.
103      * 
104      * @param introspectedTable
105      */
106     void initialized(IntrospectedTable introspectedTable);

108     /**
109      * This method is called after all the setXXX methods are called, but before
110      * any other method is called. This allows the plugin to determine whether
111      * it can run or not. For example, if the plugin requires certain properties
112      * to be set, and the properties are not set, then the plugin is invalid and
113      * will not run.
114      * 
115      * @param warnings
116      *            add strings to this list to specify warnings. For example, if
117      *            the plugin is invalid, you should specify why. Warnings are
118      *            reported to users after the completion of the run.
119      * @return true if the plugin is in a valid state. Invalid plugins will not
120      *         be called
121      */
